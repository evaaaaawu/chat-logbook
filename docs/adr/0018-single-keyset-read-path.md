# The chat list has a single keyset read path

The chat list reads through **one** path — the ATTACH keyset-paginated query of [ADR-0017](0017-cross-store-pagination-uses-attach.md) — for every case: both time axes (createdAt, updatedAt), both directions (asc, desc), the Title axis, the main view and Trash, filtered or not. There is no full-load fallback. The full-list client path (`useChats`, which fetched every Chat in one request and sorted/filtered/counted in the browser) is retired once the last axis migrates. This resolves the half-and-half state #129 shipped, where only the main view's descending time sorts paged server-side and everything else silently loaded the whole list.

The reason is consistency and scale. Two read paths meant every cross-cutting concern — sort, filter, counts, live updates — had to be built twice and kept in agreement, and the full-load half reintroduced the very cost the pipeline exists to remove (the unfiltered list materializing all Chats in JS; see the benchmark in ADR-0017). A single path makes filtering ([#130](https://github.com/evaaaaawu/chat-logbook/issues/130)) and counts ([#131](https://github.com/evaaaaawu/chat-logbook/issues/131)) uniformly server-side, with no client/server divergence to reconcile.

## Considered options

- **Keep a full-load fallback for the awkward cases (Title, ascending time, Trash).** Cheapest short-term: those axes are derived or cross-store and harder to keyset, and Title in particular needs a denormalized collation key. Rejected: it permanently splits every list concern across two code paths, and the full-load path does not hold at ~50,000 Chats — the problem the epic was opened to fix. "Bounded client load for Title" was the soft version of this and was also rejected, for the same consistency reason (see [#144](https://github.com/evaaaaawu/chat-logbook/issues/144)).
- **Single keyset read path, full-load retired.** One path, one place for each concern, page-bounded latency everywhere. Chosen. The cost is real work pushed onto the hard axes (notably the Title collation key) and a bounded-window requirement on the client, both accepted below.

## Consequences

- **Title must go full keyset.** Title is no longer allowed to stay a client-side sort. Because it is cross-store and locale-collated and SQLite has no native locale collation, this forces a denormalized, collatable sort key maintained across both write paths — the decision captured in [#144](https://github.com/evaaaaawu/chat-logbook/issues/144) and implemented in [#146](https://github.com/evaaaaawu/chat-logbook/issues/146).
- **The client window must be bounded.** With no full load, the loaded window grows as the user scrolls and the frozen-sort layer re-sorts it every render. At scale this needs page eviction (drop far-offscreen pages, re-fetch on scroll-back) so memory and per-render cost stay bounded — folded into [#132](https://github.com/evaaaaawu/chat-logbook/issues/132).
- **Counts and filtering are server-side only.** Facet counts and the list total come from SQL aggregation, not browser tallying ([#131](https://github.com/evaaaaawu/chat-logbook/issues/131)); Project/Tag filtering moves into the keyset query ([#130](https://github.com/evaaaaawu/chat-logbook/issues/130)). The client no longer holds the full set to derive either.
- **No new isolation relaxation.** This decision rides on the read-only, one-directional ATTACH that ADR-0017 already accepted (Archive reads Metadata; no writes cross). It does not relax [ADR-0001](0001-four-store-split.md) any further.
- **`useChats` is deleted, not just bypassed.** The full-load hook is removed when [#146](https://github.com/evaaaaawu/chat-logbook/issues/146) lands and nothing calls it. Until then it still serves the not-yet-migrated axes, so the migration is incremental, not a flag-day.
