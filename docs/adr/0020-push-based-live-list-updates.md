# Live list updates are a server push that signals, not a payload that replaces

The chat list stays current through a **server push that signals a change**, not a periodic refetch and not a stream of Chat rows. While the app runs, the server holds a Server-Sent Events channel (`GET /api/chats/stream`) open per client. After each ingest pass the file watcher settles, the server pushes one `changed` event. The client responds by re-reading its loaded window head through the existing keyset query — the same query that already applies the active sort and filter server-side ([ADR-0018](0018-single-keyset-read-path.md)) — and merges the result into the window, holding frozen order. This replaces the 4-second `setInterval` poll that [#129](https://github.com/evaaaaawu/chat-logbook/issues/129)'s window model shipped with.

The reason is that the poll was both too slow and unbounded. A 4-second timer means a live conversation lags by up to four seconds, and the timer fires whether or not anything changed — every loaded window, for every idle tab, on every tick. A watcher-driven push fires only on real change and arrives as fast as ingestion settles. Pushing a lightweight signal (rather than the changed rows themselves) keeps sort and filter on the server, where the rest of the pipeline already put them: the client never re-derives which rows match the active filter or where a new row sorts, so there is no second copy of that logic to keep in agreement.

## Considered options

- **Keep the 4s poll.** Cheapest — no new transport. Rejected: up-to-4s latency on a live chat, and a window-sized refetch on every tick for every client regardless of change. The poll also carried the limit-cap hazard fixed in [#147](https://github.com/evaaaaawu/chat-logbook/issues/147).
- **Push the changed Chat rows themselves (a delta of rows + sort keys), merged client-side.** Most literal reading of "push change events." Rejected: to place a pushed row the client must evaluate the active Project/Tag filter and the sort position itself — re-introducing a browser-side copy of the filtering and ordering that ADR-0018 deliberately made server-only. More surface, two sources of truth, and a correctness hazard when the two drift.
- **Push a lightweight `changed` signal; client reconciles its window head through the keyset query.** Chosen. The event is the trigger; the keyset query stays the single, server-side source of order and filter. "No full refetch" holds because the client re-reads only its bounded window head (≤ `MAX_PAGE_LIMIT`), not the whole list, and merges in place.

## Consequences

- **Filtering and sort stay server-side.** A pushed event respects the active Project/Tag filter and sort for free: the reconcile goes through the same `listChatsPage` the window already uses, with the active filter attached. No client-side filter evaluation is added ([ADR-0018](0018-single-keyset-read-path.md), [#130](https://github.com/evaaaaawu/chat-logbook/issues/130)).
- **Frozen order is preserved by the existing layer.** The reconcile feeds `mergeWindow` + the frozen-sort anchor, so an existing row updates in place and a new row slots into its sorted position without reordering the rows above it — the same path the poll used, now push-driven.
- **The channel must recover on its own.** SSE connections drop (sleep, proxy idle timeout, network blips). The client reconnects with backoff, and a low-frequency reconcile runs as a floor so a missed event cannot leave the window stale indefinitely. A periodic heartbeat from the server surfaces a dead socket promptly.
- **One in-process hub, no new store.** The push path is a process-local publish/subscribe hub the watcher publishes to and each stream connection subscribes to. It holds no state across restarts and touches none of the five stores ([ADR-0001](0001-four-store-split.md)); a fresh connection simply reconciles on connect.
- **The loaded window is now bounded.** With the poll gone, the remaining unbounded cost is the loaded window itself growing on deep scroll. Page eviction (drop far-offscreen pages, re-fetch on scroll-back) bounds memory and per-render re-sort, completing the move ADR-0018 anticipated. Tracked in the same issue ([#132](https://github.com/evaaaaawu/chat-logbook/issues/132)).
