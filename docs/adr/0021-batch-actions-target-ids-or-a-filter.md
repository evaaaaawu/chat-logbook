# Batch actions target either an explicit id set or the live filter, not a fetched id list

Batch operations over the Selection — batch Tag add/remove, batch Move to Trash, batch Restore — accept a request body that is **either** `{ chatIds: [...] }` (an explicit set the user checked) **or** `{ filter: { projects, tags, tagMode, trashedOnly }, excludeIds: [...] }` (every Chat matching the active filter, minus a small exclusion set). The server resolves each branch to the same set of internal Chat ids and applies the mutation in a single transaction: the `filter` branch reuses [`buildFilterClauses`](../../api/src/list-filter.ts) ([ADR-0016](0016-combined-filtering-app-level-intersection.md), [ADR-0017](0017-cross-store-pagination-uses-attach.md)) as an `... WHERE id IN (SELECT ... <filter>) AND id NOT IN (<excludeIds>)` subquery, so the write is one SQL pass and never enumerates ids over the wire.

The reason is the list is server-paginated with a bounded window ([ADR-0020](0020-push-based-live-list-updates.md)): the client only ever holds one page. "Select all" therefore means "all Chats matching the current filter" — a set the client cannot enumerate, and which may be thousands of rows. Sending that as an id list would mean fetching every id first (a full scan the pagination model exists to avoid) and shipping thousands of ids up per action. Passing the filter instead lets the server compute the target set with the same indexed predicate the list already uses, applied directly inside the write. The `excludeIds` set stays small by construction — it is only the rows the user unchecked after selecting all — so "all matching except these" costs one bounded `NOT IN`.

## Considered options

- **Always send an explicit id list.** Simplest contract — one code path, no filter branch. Rejected: it cannot express select-all-matching without first materializing every matching id on the client, which defeats [ADR-0020](0020-push-based-live-list-updates.md)'s bounded window and puts a full candidate scan back on the read path before every large batch.
- **A dedicated "apply to all matching filter X" endpoint, separate from the id-list one.** Two endpoints, clean separation. Rejected as premature: the two share the same mutation and transaction; a single endpoint with a discriminated body keeps the write logic in one place. The `filter`-vs-`chatIds` union _is_ the seam.
- **One endpoint, `chatIds` or `filter`+`excludeIds`.** Chosen. The filter branch reuses the existing predicate builder, so the batch write can never disagree with what the list showed as matching.

## Consequences

- **One transaction per batch, whichever branch.** `assignTagsBatch` does `INSERT ... ON CONFLICT DO NOTHING` for adds and one `DELETE ... WHERE chat_id IN (...) AND tag_id IN (...)` for removes; batch Trash/Restore flip the Metadata visibility flag over the resolved id set. No per-Chat round-trip, no N-query fan-out.
- **The filter branch and the read path share `buildFilterClauses`.** A Chat is mutated exactly when it would have appeared in the filtered list — the two can't drift, because they are the same predicate. `tagMode` (all/any) flows through unchanged.
- **Trash and Restore are the only batch mutations for now; Purge is deliberately excluded.** Purge is the sole path that deletes Archive rows ([ADR-0002](0002-never-cascade-delete-archive.md)), is not yet implemented anywhere, and carries an irreversible select-all-matching hazard (permanently deleting thousands of Chats from one gesture). It is left out of this ADR entirely and will be designed with its own confirmation model when the Purge feature is taken up.
- **Removal is server-authoritative; optimism is a local convenience where a client seam already exists.** Local SQLite round-trips are sub-millisecond, so the client applies the batch and then reloads its window head and the facet counts through the existing `reload()` + `counts.reload()` path. The client never re-derives `buildFilterClauses` to decide who leaves the list — membership stays the server's answer. Two wrinkles the implementation of batch Move to Trash (#161) surfaced, corrected here from this ADR's first draft (which assumed a plain reload would drop the mutated rows):
  - The paginated read hook's window refresh is **grow-only** — it updates fields for Chats the server still returns and keeps any loaded row the refetch no longer covers, so background ingestion never yanks a row out from under the reader. A plain reload therefore cannot, on its own, drop a row that a mutation moved out of the view.
  - **Batch Trash/Restore** ride the client's existing frozen-sort `includes` predicate (`!isDeleted` for the main list), so flipping the local `isDeleted` flag over the id set is a cheap, drift-free optimistic pass that removes (or restores) the rows at once — exactly what single-Chat Trash already does. This is used, not because optimism is preferred, but because it is the only prompt way to drop the rows given the grow-only refresh.
  - **Batch Tag add/remove** has no such client seam: the main list's `includes` predicate is `!isDeleted` only, and Tag/Project filtering lives in the server keyset query ([ADR-0016](0016-combined-filtering-app-level-intersection.md)), not on the client. So the chip update flows through `reload()` (a still-listed Chat's chips refresh in place), and dropping a Chat that a Tag change moved out of an active filter relies on the **reconciling user-reload** ([#176](https://github.com/evaaaaawu/chat-logbook/issues/176)) rather than any client-side filter logic. A narrow optimistic pass may update just the visible `chat.tags` chip set for instant feel — an id-set operation resolved from the client's tag catalog — but never the filter membership.
  - Batch Tag and batch Trash/Restore both surface an Undo toast that replays the inverse batch (add↔remove, trash↔restore) through the same endpoint.
